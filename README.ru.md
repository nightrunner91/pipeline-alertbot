## Обзор проекта

**Pipeline Alertbot** - это лёгкий, готовый к продакшену ретранслятор вебхуков, который доставляет события GitLab CI/CD пайплайнов в групповые чаты Telegram. Бот принимает вебхуки от GitLab, форматирует их в читаемые уведомления и отправляет в указанные Telegram-чаты в реальном времени. Бот поддерживает **многорежимную работу**, позволяя отслеживать множество GitLab-проектов и маршрутизировать уведомления в разные Telegram-группы.

## Возможности

- **Мониторинг множества репозиториев** - Отслеживайте множество GitLab-проектов и направляйте уведомления в разные Telegram-чаты
- **Кастомные имена проектов** - Заменяйте имена проектов из GitLab на собственные отображаемые имена в уведомлениях
- **Индивидуальные стили для каждого репозитория** - Каждый репозиторий может использовать свой формат уведомлений (card, tree или minimal)
- **Уведомления с учётом этапов** - Заголовки показывают название этапа пайплайна (например, `Running [build]`, `Failed [deploy]`)
- **Фильтрация уведомлений для каждого репозитория** - Управляйте тем, какие статусы вызывают уведомления для каждого этапа через `notifyRules`
- **Ссылки развёртывания для каждого репозитория** - Добавляйте кнопки действий (например, "Open Site", "View Swagger") в уведомления через `deployLinks`
- **Уведомления в реальном времени** - Мгновенные оповещения о запуске, успешном завершении, ошибке и отмене пайплайнов
- **Интерактивные кнопки** - Прямые ссылки на пайплайн, коммит и репозиторий прямо в Telegram
- **Инструмент конфигурации** - Управляйте конфигурацией репозиториев в читаемом JS-файле и генерируйте JSON для env
- **Безопасная валидация вебхуков** - Проверка входящих запросов по индивидуальным секретным токенам для каждого репозитория
- **Гибкая настройка порта** - Привязка к `0.0.0.0` с настраиваемым портом через переменную `PORT`
- **Надёжная обработка ошибок** - Глобальные обработчики необработанных исключений и отклонённых промисов предотвращают тихие аварийные завершения
- **Эндпоинт диагностики** - Эндпоинт проверки работоспособности, сообщающий статус конфигурации без раскрытия секретов
- **Безопасный запуск** - Бот инициализируется независимо от вебхук-сервера; отсутствие учётных данных не блокирует запуск сервера

## Начало работы

### Требования

- Node.js >= 18
- npm (поставляется вместе с Node.js)
- Токен Telegram-бота от [@BotFather](https://t.me/botfather)
- Один или несколько ID Telegram-групп/каналов для уведомлений
- Доступ к настройкам вебхуков в проектах GitLab

### Установка

1. Клонируйте репозиторий:

   ```bash
   git clone https://github.com/your-org/pipeline-alertbot.git
   cd pipeline-alertbot
   ```

2. Установите зависимости:

   ```bash
   npm install
   ```

3. Создайте файл `.env`:

   ```bash
   cp .env.example .env   # или создайте .env вручную
   ```

### Быстрый старт

1. Заполните `.env` необходимыми учётными данными (см. [Конфигурация](#конфигурация)).
2. Запустите приложение:

   ```bash
   npm start
   ```

3. Настройте вебхуки GitLab, указав URL вашего сервера (см. [Настройка вебхука GitLab](#настройка-вебхука-gitlab)).
4. Запустите пайплайн в GitLab и наблюдайте за появлением уведомления в Telegram.

## Конфигурация

### Переменные окружения

Создайте файл `.env` в корне проекта со следующими переменными:

| Переменная | Обязательная | По умолчанию | Описание |
|---|---|---|---|
| `TELEGRAM_BOT_TOKEN` | Да | -- | Токен бота, полученный от [@BotFather](https://t.me/botfather) |
| `REPOSITORY_CONFIG` | Да | -- | JSON-массив, сопоставляющий проекты GitLab с Telegram-чатами (см. ниже) |
| `ALERT_STYLE` | Нет | `card` | Глобальный запасной формат сообщений: `card`, `tree` или `minimal` |
| `PORT` | Нет | `3000` | Основной порт вебхук-сервера |

> [!IMPORTANT]
> Никогда не коммитьте файл `.env`. Он указан в `.gitignore` и должен существовать только локально или в хранилище секретов вашей платформы развёртывания.

### Многорежимная работа

Переменная `REPOSITORY_CONFIG` определяет, какие проекты GitLab отслеживать и куда отправлять их уведомления. Это JSON-массив записей репозиториев:

```json
[
  {
    "projectId": 123,
    "projectName": "Мой крутой API",
    "chatId": "-1001234567890",
    "secret": "секрет-для-репозитория-1",
    "style": "card"
  },
  {
    "projectId": 456,
    "projectName": "Фронтенд",
    "chatId": "-1009876543210",
    "secret": "секрет-для-репозитория-2",
    "style": "tree"
  }
]
```

**Поля для каждой записи репозитория:**

| Поле | Обязательное | Описание |
|---|---|---|
| `projectId` | Да | ID проекта GitLab (числовой, находится в Settings > General) |
| `projectName` | Нет | Кастомное отображаемое имя для уведомлений (заменяет имя проекта из GitLab) |
| `chatId` | Да | ID Telegram-чата/группы для отправки уведомлений |
| `secret` | Нет | Секретный токен вебхука (должен совпадать с настройками вебхука в GitLab) |
| `style` | Нет | Стиль уведомлений: `card`, `tree` или `minimal` (резервный -- `ALERT_STYLE`) |
| `notifyRules` | Нет | Правила фильтрации уведомлений для каждого этапа (см. [Правила уведомлений](#правила-уведомлений)) |
| `deployLinks` | Нет | Кастомные кнопки действий для уведомлений (см. [Ссылки развёртывания](#ссылки-развёртывания)) |

В файле `.env` JSON должен быть записан в **одну строку**:

```env
REPOSITORY_CONFIG=[{"projectId":123,"projectName":"Мой API","chatId":"-1001234567890","secret":"секрет1","style":"card"},{"projectId":456,"projectName":"Фронтенд","chatId":"-1009876543210","secret":"секрет2","style":"tree"}]
```

### Правила уведомлений

Поле `notifyRules` позволяет контролировать, какие статусы пайплайна вызывают уведомления для каждого этапа. Это полезно для уменьшения шума -- например, получать оповещения только о тестах или успешных развёртываниях.

```json
{
  "projectId": 123,
  "projectName": "Мой API",
  "chatId": "-1001234567890",
  "notifyRules": {
    "build":  { "send": ["success", "failed", "running"], "ignore": ["canceled", "pending", "manual"] },
    "deploy": { "send": ["success", "failed"], "ignore": [] },
    "test":   { "send": ["failed"], "ignore": [] }
  }
}
```

**Как это работает:**

- `send` -- это **белый список** -- только эти статусы вызывают уведомление для данного этапа.
- `ignore` -- это **чёрный список** -- эти статусы пропускаются, даже если они должны были быть отправлены.
- Если этап **не указан** в `notifyRules`, все его события отправляются (обратная совместимость).
- Если `notifyRules` **отсутствует** полностью, все события отправляются (поведение по умолчанию).
- `send` **имеет приоритет**: если статус находится и в `send`, и в `ignore`, он будет отправлен.
- Допустимые статусы: `success`, `failed`, `running`, `canceled`, `pending`, `manual`.
- Название этапа извлекается из полезной нагрузки GitLab (`detailed_status.context` или первая запись в `stages`). Если этап не может быть определён, он считается `"unknown"` -- если `"unknown"` нет в `notifyRules`, событие отправляется.

### Ссылки развёртывания

Поле `deployLinks` добавляет кастомную кнопку на всю ширину в нижней части клавиатуры уведомления. Каждый этап может иметь свою ссылку с кастомным именем:

```json
{
  "projectId": 123,
  "projectName": "Мой API",
  "chatId": "-1001234567890",
  "deployLinks": {
    "build":  { "url": "", "name": "" },
    "deploy": { "url": "https://my-api.example.com", "name": "Открыть сайт" },
    "test":   { "url": "https://my-api.example.com/swagger", "name": "Просмотр Swagger" }
  }
}
```

- Каждый этап может иметь свою ссылку с `url` и `name`.
- Ссылки необязательны -- если у этапа нет `url`, кнопка не отображается.
- Кнопка появляется на отдельной строке в самом низу, после кнопок Pipeline/Commit/View repository.
- Если `deployLinks` не настроен для репозитория или текущего этапа, поведение не меняется.

### Стили уведомлений

Поле `style` (для каждого репозитория) или `ALERT_STYLE` (глобальный резервный) управляет тем, как уведомления о пайплайнах выглядят в Telegram:

| Стиль | Описание |
|---|---|
| `card` | Чистая карточка с жирными заголовками и моноширинными значениями |
| `tree` | Структурированный список с древовидными соединителями (`├─` / `└─`).|
| `minimal` | Компактный однострочный формат с встроенными метаданными |

Все стили включают интерактивную клавиатуру с кнопками-ссылками на пайплайн, коммит и репозиторий. Когда `deployLinks` настроен, дополнительная кастомная кнопка появляется внизу.

**Заголовки с учётом этапов:** Все стили уведомлений теперь включают название этапа пайплайна в строке заголовка:

```
🔄 Running [build]
❌ Failed [deploy]
✅ Passed [test]
```

Название этапа извлекается из полезной нагрузки GitLab (`detailed_status.context` или первая запись в `stages`). Если ни одно из них недоступно, используется резервный вариант `[pipeline]`.

### Инструмент конфигурации

Вместо написания встроенного JSON в `.env`, вы можете использовать инструмент конфигурации для более удобного рабочего процесса:

1. Отредактируйте `config/repos.config.js` -- читаемый формат JS-объекта для всех ваших репозиториев.
2. Сгенерируйте JSON для env:

   ```bash
   # Вывести JSON в stdout
   npm run config:build

   # Вывести в формате REPOSITORY_CONFIG=
   npm run config:build -- --env

   # Записать напрямую в .env
   npm run config:write
   ```

> [!NOTE]
> Директории `config/` и `scripts/` исключены из git через `.gitignore`. Только сгенерированное значение `.env` должно коммититься или храниться в секретах вашей платформы развёртывания.

### Настройка вебхука GitLab

Настройте вебхук **для каждого репозитория**, который вы хотите отслеживать:

1. Откройте ваш проект GitLab и перейдите в **Settings > Webhooks**.
2. Установите **URL** на ваш развёрнутый эндпоинт:

   ```
   https://your-domain.com/api/webhook/gitlab
   ```

3. Установите **Secret token** в значение `secret` из соответствующей записи в `REPOSITORY_CONFIG`.
4. В разделе **Trigger** отметьте **Pipeline events**.
5. Оставьте **Enable SSL verification** включённым для продакшен-развёртываний.
6. Нажмите **Add webhook** и протестируйте через выпадающее меню **Test > Pipeline events**.

> [!NOTE]
> Все репозитории используют один и тот же эндпоинт вебхука. Бот маршрутизирует уведомления в правильный Telegram-чат на основе `projectId` во входящей полезной нагрузке.
>
> Бот обрабатывает только события пайплайнов со статусами: `running`, `success`, `failed` и `canceled`. Остальные события подтверждаются, но игнорируются.

### Развёртывание

Приложение можно развернуть на любой платформе, поддерживающей Node.js (Railway, Render, Fly.io, Docker, VPS и т.д.):

1. Установите необходимые переменные окружения в конфигурации вашей платформы:
   - `TELEGRAM_BOT_TOKEN` -- токен вашего бота
   - `REPOSITORY_CONFIG` -- JSON-массив сопоставлений репозиториев в одну строку
   - `ALERT_STYLE` -- опциональный глобальный резервный стиль (по умолчанию: `card`)
   - `PORT` -- порт, назначенный вашей платформой (по умолчанию: `3000`)
2. Установите команду запуска:

   ```bash
   node src/index.js
   ```

3. Разверните приложение. Сервер привяжется к `0.0.0.0` на указанном порту.

> [!NOTE]
> Убедитесь, что ваша платформа делает эндпоинт вебхука общедоступным через HTTPS. GitLab требует SSL-верификацию для вебхуков в продакшене.

## Использование

### Локальный запуск

```bash
npm start
```

Ожидаемый вывод:

```json
{"level":"INFO","timestamp":"2026-05-22T12:00:00.000Z","message":"Telegram bot initialized successfully","context":{}}
{"level":"INFO","timestamp":"2026-05-22T12:00:00.000Z","message":"Loaded 2 repository configuration(s)","context":{}}
{"level":"INFO","timestamp":"2026-05-22T12:00:00.000Z","message":"Starting webhook server...","context":{}}
{"level":"INFO","timestamp":"2026-05-22T12:00:00.000Z","message":"Webhook server listening on 0.0.0.0:3000","context":{}}
```

### Эндпоинт вебхука

- **URL**: `POST /api/webhook/gitlab`
- **Заголовки**: `X-Gitlab-Token: <секрет-репозитория>`
- **Тело**: Полезная нагрузка события пайплайна GitLab (JSON)
- **Ответ**: `200 OK` при успехе, `401 Unauthorized` если секрет не совпадает, `404` если конфигурация репозитория не найдена, `500` если бот не инициализирован

### Эндпоинт диагностики

- **URL**: `GET /`
- **Ответ**: JSON-объект со статусом сервера и наличием конфигурации (секреты не раскрываются)

```json
{
  "status": "running",
  "env": {
    "PORT": 3000,
    "HAS_TELEGRAM_BOT_TOKEN": true,
    "HAS_TELEGRAM_GROUP_ID": false,
    "HAS_GITLAB_WEBHOOK_SECRET": false,
    "HAS_REPOSITORY_CONFIG": true,
    "ALERT_STYLE": "card"
  },
  "botInitialized": true,
  "registeredRepositories": 2
}
```

| `npm run test:webhooks` | Интеграционные тесты вебхуков с примерами полезных нагрузок |
| `npm run test:security` | Тесты валидации безопасности (проверка токена, санитизация входных данных) |

Тесты используют файлы фикстур из `test/fixtures/`, которые имитируют реальные полезные нагрузки событий пайплайнов GitLab.

### Добавление нового стиля уведомлений

1. Откройте `src/services/message-builder.js`.
2. Создайте новую функцию форматирования (например, `formatCompactStyle(data)`).
3. Добавьте case в оператор switch функции `buildMessage()`.
4. Добавьте соответствующие тесты в `test/test-unit.js`, используя существующую или новую фикстуру.
5. Запустите `npm test` для проверки.

### Логирование

Приложение использует структурированное JSON-логирование через `src/utils/logger.js`. Все записи журнала включают `level`, `timestamp`, `message` и опциональный `context`. Ошибки дополнительно включают поля `error` и `stack`. Логи записываются в `stdout` (INFO) и `stderr` (ERROR) для удобной интеграции с системами агрегации логов.
